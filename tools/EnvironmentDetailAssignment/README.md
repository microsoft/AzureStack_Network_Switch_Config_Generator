# Environment Detail Assignment Tool Specification

## Purpose

Standardize IP payload definitions for complex environments by consuming subnet/VLAN plans produced by the `IPAssignment` PowerShell module and an environment-specific assignment manifest. The script validates the two JSON inputs, auto-reserves core network components, and emits an authoritative inventory of IP allocations in Markdown (default), CSV, or JSON formats.

## High-Level Workflow

1. **Collect Inputs**
   - Import the subnet/VLAN catalog generated by `IPAssignment` (e.g., `IPAssignment-Example.json`).
   - Import the Environment Detail manifest describing devices, roles, and overrides per subnet.
2. **Validate**
   - Ensure every environment subnet name maps to a subnet in the `IPAssignment` dataset (case-insensitive exact match).
   - Confirm uniqueness of network names and IP assignments.
   - Enforce presence of required metadata.
3. **Enrich & Allocate**
   - Auto-reserve gateway, top-of-rack (ToR), network, and broadcast addresses per subnet when not explicitly set.
   - Apply manifest overrides for device roles, host pools, and reserved ranges.
   - Track derived assignments and flag conflicts.
4. **Emit Outputs**
   - Produce Markdown tables (default), CSV files, or JSON documents summarizing all allocations.
   - Include optional metadata columns when supplied.

## Script Contract

| Aspect             | Detail                                                                                                                       |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **Implementation** | PowerShell 7+ script (`EnvironmentDetailAssignment.ps1`) with functions exported via a module wrapper if desired.            |
| **Invocation**     | Manual execution through PowerShell; pipeline integration supported by parameterizing paths and format switches.             |
| **Logging**        | Leverage `Write-Verbose` and `Write-Debug` for traceability. Emit structured error records for failures.                     |
| **Config**         | Two JSON inputs provided via parameters; script resolves a default manifest located alongside the script when not specified. |

## Input Schemas

### 1. IP Address Plan (from `IPAssignment`)

- Array of objects similar to [`IPAssignment-Example.json`](../EnvironmentDetailAssignment/IPAssignment-Example.json).
- Required fields: `Name`, `Vlan`, `Subnet`, `Prefix`, `Network`, `Broadcast`, `FirstHost`, `EndHost`, `UsableHosts`, `Category`.
- The script treats `Name` as the authoritative key to align with the environment manifest.

### 2. Environment Detail Manifest (new)

Goal: Describe every device, service, or reserved range per subnet. Example structure:

```json
{
  "metadata": {
    "environment": "contoso-prod-east",
    "version": "2025.09.0",
    "owner": "network-ops@contoso.com",
    "generated": "2025-09-26"
  },
  "subnets": [
    {
      "name": "csu-edge-transport-compute",
      "vlan": "203",
      "devices": [
        { "role": "tor", "hostname": "csu-tor-01", "ip": "auto" },
        { "role": "tor", "hostname": "csu-tor-02", "ip": "auto" },
        {
          "role": "load-balancer",
          "hostname": "csu-flx-lb01",
          "ip": "192.168.0.10"
        },
        {
          "role": "application",
          "hostname": "csu-edge-01",
          "ip": "pool",
          "count": 4
        }
      ],
      "notes": "HSRP pair on VLAN 203"
    }
  ]
}
```

#### Device Entry Semantics

- `ip` values can be:
  - Exact IPv4 address (validated within subnet bounds).
  - `"auto"` to request automatic assignment following business rules.
  - `"pool"` accompanied by `count` and optional `startOffset` to reserve consecutive addresses.
- Optional fields (`role`, `hostname`, `description`, `owner`, `lifecycle`, `tags`) surface as metadata columns.

## Allocation Logic & Business Rules

- **Subnet Presence**: Every manifest subnet name must exist in the IP plan. Absence triggers a terminating error.
- **Default Reservations** (applied when not explicitly assigned):
  - Network address (`Network` field).
  - Gateway: first usable IP (`FirstHost`).
  - ToR pairs: sequential IPs immediately following the gateway when two or more ToRs specified. For a single ToR request, reuse gateway.
  - Broadcast address (`Broadcast`).
- **Auto Assignment Order**:
  - Reserve defaults.
  - Process explicit `ip` values.
  - Fulfill `pool` reservations in ascending order, skipping already allocated addresses.
  - Assign remaining `auto` devices sequentially after the highest reserved address, respecting subnet bounds.
- **Collision Detection**: Duplicate IP requests or allocations outside `FirstHost`..`EndHost` cause errors with detailed context.
- **Output Completeness**: Every subnet row includes network, gateway, ToR allocations (if any), broadcast, and each device entry.

## Output Specifications

### Markdown (default)

- One table per subnet with the columns: `VLAN ID`, `VLAN Name`, `Device`, `Role`, `IP Address`, `Subnet Mask`, `CIDR`, `Network`, `Broadcast`, plus optional metadata columns when present.
- Provide a summary section listing total allocations and any skipped/auto-generated notes.

### CSV

- File name pattern: `<environment>-ip-assignments.csv`.
- Required columns: `VlanId`, `VlanName`, `Device`, `Role`, `IpAddress`, `SubnetMask`, `Cidr`, `Network`, `Broadcast`.
- Optional columns appended when metadata exists (e.g., `Owner`, `Lifecycle`, `Notes`).

### JSON

- Mirrors CSV schema as an array of objects using camelCase keys.
- Example entry:

```json
{
  "vlanId": "203",
  "vlanName": "csu-edge-transport-compute",
  "device": "csu-tor-01",
  "role": "tor",
  "ipAddress": "192.168.0.2",
  "subnetMask": "255.255.255.192",
  "cidr": 26,
  "network": "192.168.0.0",
  "broadcast": "192.168.0.63",
  "owner": "network-ops",
  "notes": "HSRP primary"
}
```

## Logging & Diagnostics

- Use `Write-Verbose` for allocation steps (subnet matches, auto assignments, metadata enrichment).
- Use `Write-Debug` for low-level trace (e.g., pointer offsets, pool expansions).
- On validation failures, throw terminating errors enriched with `ErrorCategory` and include subnet/device context.
- Summaries of warnings (e.g., unused manifest entries) should display regardless of verbosity.

## Metadata Recommendations

To aid downstream consumers, include the following optional metadata fields in the manifest and propagate them to outputs:

- `owner` (team or contact responsible for the IP block)
- `lifecycle` (e.g., `production`, `staging`, `lab`)
- `application` or `service` identifiers
- `dependents` (list of systems relying on the device)
- `changeTicket` or `workItem` references
- `notes` for implementation instructions
- `tags` array for arbitrary classification (exported as semicolon-delimited string in CSV/Markdown)

## Error Handling Matrix

| Scenario                                | Response                                               |
| --------------------------------------- | ------------------------------------------------------ |
| Subnet in manifest missing from IP plan | Terminating error specifying subnet name.              |
| Duplicate subnet names in manifest      | Terminating error listing duplicates.                  |
| IP outside usable range                 | Terminating error with offending IP and subnet bounds. |
| Duplicate IP allocation                 | Terminating error noting both competing devices.       |
| Unsupported `ip` directive              | Terminating error referencing device entry.            |
| Output path unwritable                  | Terminating error with original exception details.     |

## Implementation Notes

- Target PowerShell 7 or later to leverage cross-platform compatibility and modern language features (e.g., classes, `ForEach-Object -Parallel`).
- Provide a companion unit-test suite (e.g., Pester) that validates manifest parsing, auto-allocation logic, collision detection, and output formatting across Markdown/CSV/JSON modes.
- Build helper functions: `Get-SubnetContext`, `Get-NextAvailableIp`, `Resolve-DeviceAssignments`, `Export-IpAssignments`.
- Prefer `[ipaddress]` type accelerators for comparisons and range checks.
- Generate subnet mask via `System.Net.IPNetwork` or custom calculation from prefix.
- Ensure script returns objects (not just writes files) to support pipelines (`-PassThru` switch optional).

## Future Enhancements (Optional)

- Add pipeline mode that ingests subnet JSON from stdin.
- Support IPv6 allocations via dual-stack manifests.
- Introduce HTML output for documentation portals.
- Integrate with the network config generator to auto-populate switch templates.
